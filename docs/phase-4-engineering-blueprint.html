Engineering Blueprint — Phase 4 · v1.0

The engineering handbook for NightCravings.

Translates Phases 1–3 into a buildable system. Optimized in this order when a tradeoff forces a choice: correctness of money/inventory state → security → maintainability → performance → developer experience. Written so any senior engineer can pick up any section and start implementing without a clarifying call.

Depends on Phase 1 + 2 + 3 Status Ready for implementation Date 11 Jul 2026

Self-Review — architectural decisions challenged before writing this blueprint

Six defaults an engineer reaches for on autopilot were tested against this system's actual scale (one hostel today, thousands eventually) and actual risk profile (real money, manual verification, a solo non-technical operator).

Risk
Microservices / multi-repo split "for future scale." At current and near-term scale this adds deployment complexity and cross-service consistency bugs for zero present benefit.
Decision
Single Next.js application + single Supabase project, modular monolith internally (feature folders, §20). Extraction into services is deferred until a specific bottleneck (e.g. search, or a payments-webhook fan-out) actually demands it — see §22 for the exact trigger conditions.
Risk
Hand-rolled Sessions and Devices tables, as literally listed in the brief, would duplicate what Supabase Auth already manages correctly (refresh tokens, session expiry, revocation) and become a security liability the moment it drifts from the real auth state.
Decision
Supabase Auth owns sessions natively. A Devices table still exists, but scoped narrowly to push-notification tokens per device — not session state (§3, §4).
Risk
Full-text search infrastructure (Algolia, Meilisearch, Postgres tsvector with a ranking service) is the reflexive answer to "Search Architecture" — and is over-engineering for a catalogue of a few hundred SKUs per hostel.
Decision
Client-side fuzzy index (confirmed in Phase 2 §2) remains correct through multi-hostel scale, since search is always scoped to one hostel's catalogue at a time. Server-side search is revisited only at the point a single hostel's catalogue exceeds ~2,000 SKUs — a trigger condition stated explicitly in §9, not a vague "later."
Risk
Payment screenshot storage growth is unbounded by default — every order's proof-of-payment image, retained forever, is a real, compounding storage cost and a data-liability surface with no offsetting benefit past the audit window.
Decision
A storage lifecycle policy (§7) auto-transitions screenshots to cold/archived storage after 90 days and hard-deletes after 1 year, aligned to a stated audit retention requirement rather than "keep everything."
Risk
Multi-tenant Row Level Security is the single highest-consequence bug class in this stack — a missing tenant_id check on one policy silently leaks one hostel's orders/customers to another's owner console.
Decision
Every tenant-scoped table carries a non-nullable hostel_id, RLS policies are generated from one shared helper predicate (not hand-written per table), and a pgTAP cross-tenant-leak test suite is a hard CI gate before any RLS change merges (§6, §19).
Risk
Supabase Realtime has a per-project concurrent-connection ceiling that a naive "every screen subscribes to everything" pattern hits earlier than expected once hundreds of hostels are live simultaneously.
Decision
Channels are scoped per-hostel, not global (§10); the connection budget and a sharding-by-region multi-project strategy for franchise scale are stated explicitly with numbers in §22 and §23, not left implicit.

Database Design

Every tenant-scoped table carries a non-nullable hostel_id uuid references hostels(id) — this single column is the entire multi-tenancy strategy and the entire RLS strategy (§6). Timestamps are created_at timestamptz default now() and, where mutable, updated_at timestamptz maintained by a shared trigger — stated once here rather than repeated per table below.

Authentication

ActorMethodWhy
CustomerPhone number + OTP (Supabase Auth phone provider)Phone is the identity students already use for UPI and WhatsApp — email login would add a field nobody wants to fill in a 30-second checkout (Phase 2).
Owner / Manager / staffEmail + password, with magic-link as an alternateA console operator logs in far less often than a customer orders, and typically from a remembered device — password is acceptable friction here in exchange for not depending on SMS deliverability for a business-critical login.
Future OAuthGoogle, reservedDeferred until a second/third staff hire makes "remembering another password" a real onboarding friction point.
  • Session management: handled entirely by Supabase Auth's JWT + refresh-token rotation — no custom session table (Self-Review). Access token short-lived (1h), refresh token long-lived (30d) for "Remember me" by default on both apps, since re-entering an OTP every visit would violate Phase 2's 30-second promise.
  • Multi-device: Supabase Auth natively supports multiple concurrent refresh tokens per user — a customer on their phone and laptop, or two staff devices on the Owner Console (Phase 3 §18), both stay signed in independently.
  • Device management UI: Profile → Devices lists active sessions by last-seen + platform with a per-device "Sign out" — reads from auth.sessions via a Supabase RPC, not a hand-maintained table.
  • Rate limiting on OTP: max 5 requests per phone number per 15 minutes, enforced at the Supabase Auth level plus an Edge Function guard — see §13.

Authorization

Two distinct models, deliberately not unified: staff authorization is role-based (RBAC); customer authorization is identity-based (a customer may only ever act on their own orders/profile — there's no "customer role" to assign).

RoleGrants
ownerAll permissions, all screens, including settings.write_payment_identity and financial exports.
managerAll except settings.write_payment_identity and reports.export_financial — day-to-day operation without irreversible-feeling changes (Phase 3 §12).
inventory_managerinventory.*, products.*; read-only elsewhere.
delivery_partnerorders.read_out_for_delivery, orders.mark_delivered only — no customer PII beyond room number and phone, no financial data at all.
customer (implicit)orders.read_own, orders.create, profile.write_own — enforced entirely by RLS predicate customer_id = auth.uid(), no role row required.

Permission checks happen twice, deliberately: once in RLS (§6) as the non-negotiable database-level boundary, and once in the UI/Server Action layer to fail fast with a good error message rather than a raw RLS-denied Postgres error reaching a user.

06

Row Level Security

One shared predicate, reused everywhere, per Self-Review: hostel_id = (select hostel_id from profile_roles where profile_id = auth.uid()) for staff, and customer_id = auth.uid() for customer-owned rows. Hand-written per-table policies are exactly the pattern that produces the "missing check on one table" leak class — so policies are generated from two Postgres functions (is_staff_of(hostel_id), owns_row(customer_id)) and every table's policy is a one-line call to one of them.

Edge cases

  • Customer with no hostel_id yet (pre-first-order): can read any hostel's public catalogue (browsing before checkout) but cannot read any order/payment data — enforced by the policy checking customer_id ownership, which simply returns no rows for a customer who's never ordered.
  • Delivery Partner reading an order: policy additionally requires status = 'out_for_delivery' — a Delivery Partner cannot see Awaiting Verification payment data even within their own hostel.
  • Cross-tenant staff account (a manager helping at two hostels once multi-hostel exists): profile_roles supports multiple rows per profile, and is_staff_of(hostel_id) checks membership, not a single denormalized field on profiles — this is precisely why that helper function exists rather than a flat column.
  • Service-role bypass: only Edge Functions running scheduled jobs (hash computation, storage lifecycle, nightly recommendation recompute) use the service-role key, and only from server-side Edge Function code that never reaches the client bundle — enumerated explicitly in §13.

09

Search Architecture

  • Indexing: a per-hostel JSON index (id, name, synonyms, category) generated on every product write via a Postgres trigger → Storage, fetched once per customer session and cached in the PWA's IndexedDB — no server round-trip per keystroke (Phase 2 §2).
  • Synonyms: the products.search_keywords array (§3), authored by the owner in Product Management (Phase 3 §5) — data-driven, not a hardcoded app-level map, so it scales per-hostel without a deploy.
  • Typo tolerance: client-side Levenshtein distance-1 matching against the cached index (a small, fast library, no server dependency).
  • Ranking: in-stock first, then personal purchase frequency (a lightweight per-customer count fetched once at session start), then hostel-wide popularity from analytics_events.
  • Caching: index is versioned by a hash of the catalogue's last-modified timestamp — the client only re-fetches when that hash changes, otherwise serves entirely from IndexedDB, offline-capable.
  • Trigger condition for server-side search: revisit (Postgres tsvector + pg_trgm, still not a separate search service) only once a single hostel's active catalogue exceeds ~2,000 SKUs, at which point a client-side index stops being reasonably small to ship.
  • Future AI search: a natural-language "I want something spicy and quick" query, reserved as a v2 Edge Function calling an embeddings model against product descriptions — explicitly not built now, since the current catalogue size makes keyword search sufficiently good.

Realtime Architecture

ChannelScopeConsumers
orders:hostel_idPer-hostel, filtered server-side by RLS-aware Realtime policyOwner Console Order Queue, Dashboard tiles

  • Never a global/unscoped channel: every subscription is hostel-scoped from day one — this is both the security boundary (paired with RLS) and the connection-budget control named in the Self-Review.
  • Conflict resolution: order status transitions are last-write-wins at the database level, but only one actor legitimately writes a given transition (the owner console for stage advances) — the actual conflict this system faces is optimistic-UI-vs-server-truth, not concurrent writers, resolved by treating the realtime event as authoritative and silently reconciling any optimistic local state that disagrees (Phase 3 §2).
  • Connection budget: Supabase's per-project concurrent Realtime connection ceiling is planned against explicitly in §22/§23 — the mitigation is one Supabase project per region/shard at franchise scale, not a single global project absorbing every hostel's connections.

13

Security

ConcernControl
AuthenticationSupabase Auth, OTP rate-limited (§4); JWT signature verified server-side on every Server Action, never trusted from client-supplied claims alone.
Authorization / RLSEvery table policy-gated (§6); no table ships with RLS disabled, including new tables — enforced by a CI check that fails the build if a migration adds a table without a corresponding policy file.
CSRFServer Actions use Next.js's built-in origin-check + action-encrypted tokens — no bespoke CSRF middleware needed.
XSSReact's default escaping everywhere; the one raw-HTML surface (product descriptions) is sanitized server-side on write (DOMPurify-equivalent) before storage, not just on render.
SQL injectionNo raw SQL string concatenation anywhere — all queries via Supabase's parameterized client or typed query builder.
Rate limitingEdge middleware rate-limits OTP requests, order creation, and payment-screenshot upload per IP + per phone number (sliding window, Upstash Redis) — specifically to blunt both brute-force OTP guessing and order-spam.
Brute forceOTP lockout after 5 failed attempts per 15 minutes; owner/manager password login lockout after 10 failed attempts per hour with an email alert.
Secrets / env varsService-role key and UPI credentials live only in Vercel encrypted env vars and Supabase Vault respectively — never in client bundles, never in the repo, verified by a pre-commit secret-scanner.
File upload validationPayment screenshots and product images: MIME-type allowlist (JPEG/PNG/WebP only), max 8MB, dimension sanity check — rejected client-side first for UX, re-validated server-side as the actual gate.
Image / malware preventionAll uploads re-encoded server-side (§7) rather than stored as-uploaded — re-encoding strips any polyglot-file attack surface, since the stored bytes are never the original upload.
Payment screenshot integrityPerceptual hash (§3, §6) plus the fraud-flag counter on the customer profile — the two mitigations named in the Self-Review, working together.
EncryptionTLS everywhere (Vercel/Supabase default); UPI ID/number encrypted at rest via Supabase Vault, decrypted only server-side when rendering the payment screen.
CSP / headersContent-Security-Policy restricting script/style/connect-src to Vercel + Supabase + Sentry/PostHog origins only; Strict-Transport-Security, X-Frame-Options: DENY, X-Content-Type-Options: nosniff set globally via Next.js middleware.

Observability

  • Logging: structured JSON logs from Edge Functions and Server Actions, shipped to Vercel's log drain; every log line carries hostel_id and request_id for traceability.
  • Error monitoring: Sentry across client, server, and Edge runtimes, with source maps for real stack traces and session replay scoped to the Owner Console (the surface where a hard-to-reproduce bug costs real operational time).
  • Tracing: Sentry's performance tracing on Server Actions and Postgres queries — flags any query exceeding 200ms as a named issue, not just an aggregate p95.
  • Health checks: a /api/health edge route checking Postgres and Storage reachability, polled by Vercel's monitoring and an external uptime check (so a Vercel-side outage is also caught).
  • Alerts: Sentry alert → Slack/email on any error affecting the order/payment path specifically (tagged separately from general UI errors) — these page the on-call engineer; everything else is triaged next business day.
  • Database monitoring: Supabase's built-in query performance dashboard, reviewed weekly against the named indexes in §12 to catch drift as the catalogue and order volume grow.

Queue: all outbound notifications are written to the notifications table first (§3) and dispatched by an Edge Function reading that table — the table is the durable queue, so a dispatch failure never loses the notification, only delays it to the next retry pass.

PWA

  • Manifest: separate manifests for the customer app and Owner Console (different name, icon, theme color) even though they share one codebase — installed as two distinct "apps" on a device, matching how the owner and a resident actually think about them.
  • Service worker caching: stale-while-revalidate for the catalogue/search index (Phase 1 §14, §9 above); network-only for checkout/payment submission — money-moving requests are never served from cache or allowed to silently queue as if they'd succeeded.
  • Offline queue: non-money-moving owner actions (Mark Delivered, stock adjustments) queue in IndexedDB when offline and sync on reconnect (Phase 3 §16, §18); customer checkout instead shows an explicit "you're offline" state rather than a false queued-success.
  • Install prompt: deferred and shown contextually after a customer's second order (proven intent) rather than on first visit, where it reads as an interruption.
  • App icons / splash: generated from the Phase 1 design tokens — warm paper background, amber mark — consistent with the brand across both installed apps.
  • Shortcuts: manifest shortcuts for "Reorder Last Order" (customer) and "Order Queue" (owner) — one long-press from the home screen icon straight to the highest-frequency action.

Coding Standards

  • TypeScript: strict mode, no any (lint error, not warning), exhaustive switch/enum handling enforced by @typescript-eslint/switch-exhaustiveness-check — critical for the order-status state machine.
  • React: Server Components by default; a component opts into 'use client' only when it needs interactivity, state, or a browser API — never as a reflex.
  • Hooks: no business logic inside components — data-fetching and mutation logic lives in features/*/hooks, components stay presentational.
  • Naming: files match their default export (OrderCard.tsx exports OrderCard); booleans read as questions (isOpen, not open); Server Actions are verbs (approvePayment, not paymentApproval).
  • Imports: absolute via the aliases in §20; no cross-feature deep imports — a feature exposes what other features need via its own index.ts.
  • Comments: only where the why isn't obvious from the code (per Phase 1's own stated principle) — never restating what a line does.
  • Error handling: Server Actions return a typed { data } | { error } result, never throw across the client/server boundary — callers handle both branches explicitly, no silent catch-and-ignore.
  • Logging: structured, never console.log left in committed code (lint-enforced) — use the shared logger that tags hostel_id/request_id automatically.
  • Validation: one Zod schema per entity, imported by both the client form and the Server Action — never two separate validation definitions that can drift.
  • Code review checklist: does this table have an RLS policy? does this mutation have a rollback/optimistic-update path? does this touch money/stock — if so, is it covered by an integration or pgTAP test? is any new UI checked against both themes and WCAG AA?

Implementation Roadmap

Sprint 1 — Foundation

Schema + RLS (§3, §6) for hostels/profiles/products/categories/orders/order_items/payments; Supabase Auth (phone OTP + email/password); base Next.js app shell with both route groups and design tokens carried over from Phase 1. Dependency: nothing downstream can start until RLS is in place and pgTAP-tested — this sprint is the one everything else depends on.

Sprint 2 — Customer ordering path

Home, Search, Category, Product Card/Sheet, Cart, Checkout (Phase 2 §1–7). Depends on: Sprint 1 schema; client-side search index generation (§9) can start in parallel once products exists.

Sprint 3 — Payment & order fulfillment

Payment upload flow (Phase 2 §8), Owner Console Order Queue + inline verification (Phase 3 §2–3), duplicate-hash Edge Function, realtime channels (§10). Depends on: Sprint 2's checkout producing real orders to verify against; this is the highest-risk sprint given the Self-Review's payment-fraud and RLS concerns, and should not be compressed.

Sprint 4 — Operations

Inventory (velocity-model low stock), Product Management, Customers, Store Settings, Notifications (Phase 3 §4–6, §8, §11, §16). Depends on: Sprint 3's order/payment data existing to compute velocity and customer stats against.

Sprint 5 — Analytics, PWA, hardening

Analytics dashboards (§7/§15), PWA install/offline (§17), full security pass (§13), Lighthouse/axe/pgTAP CI gates (§19), Reports export (Phase 3 §10). Depends on: a full sprint's worth of real order data to make Analytics meaningful; everything else in this sprint can run in parallel with Sprint 4.

Acceptance Criteria

The engineering architecture is complete when all of the following hold simultaneously:

  • Every table has a corresponding RLS policy, and the pgTAP suite passes for every cross-tenant and role-boundary case enumerated in §6.
  • The full order lifecycle (Phase 3 §15) executes end-to-end against a real Postgres instance via the integration test suite, including every branch in Phase 3 §16's edge-case table.
  • Lighthouse CI passes the Core Web Vitals budget in §12 on both the customer Home and the Owner Console Order Queue.
  • A payment can be uploaded, hashed, flagged for duplication where applicable, and approved/rejected with the resulting state reflected on both the customer and owner side within 1 second via realtime.
  • Two simultaneous devices logged into the same hostel's Owner Console see and can act on consistent realtime state, with no double-fulfillment of a single order.
  • Every environment variable and secret is confirmed absent from the client bundle via a build-time bundle-analysis check.
  • The folder structure, alias set, and lint rules in §20–21 are enforced by CI, not just documented.
